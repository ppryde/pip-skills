# Overseer Interactive Dashboard ‚Äî Design Spec

**Date:** 2026-07-11
**Status:** Approved design. Managed as overseer cards (an epic + children) ‚Äî this
spec is the pre-card reference for large work.
**Scope:** Spec 3 of 3 in the relationships/doctrine/dashboard set. Specs 1
(relationships) and 2 (doctrine ports) are merged to `main`.

## Context

Overseer renders a text view of its cards (`ledger.md`, generated by `index.py`)
and a re-injectable handoff. This adds an **interactive web board** ‚Äî a control
surface over overseer (cards) and vigil (context threshold) ‚Äî so a human can *see*
the epic tree / dependency edges / readiness / budgets and *act*: reorder,
prioritise, link (parent/deps), park, change status, expand-to-view, and set the
vigil threshold.

## Decisions already made

| Decision | Choice | Rationale |
|---|---|---|
| Interactivity | Full interactive (read-write), served web app | The user wants to reorder/prioritise/link/park/set-threshold from the board, not just view |
| Placement | **Internal** to overseer, as an isolated optional `dashboard/` subtree | Ships with overseer, but the core `scripts/`/`skills/` stay pure-stdlib and untouched; the dashboard is opt-in with its own deps |
| Tech | FastAPI backend + React/Vite frontend (committed `dist/`) | Chosen for a polished UI; the cost (Python web deps + a JS build) is *confined* to the `dashboard/` subtree, not the core |
| Write boundary | **The backend talks only to the CLIs** ‚Äî `overseer` for card reads/writes, `vigil config set` for the threshold | Preserves overseer's single-writer invariant: the CLI stays the only writer; the board never edits `.workflow/` directly |
| Reorder/prioritise | Add `order: int` + `priority: P0‚ÄìP3` to the core Card model | Cards lack these today; drag-reorder and prioritise need them (pure-stdlib core change) |
| Deployment | Local only (`127.0.0.1`), single user, no auth; committed `frontend/dist/` | Running needs only the Python deps; node/Vite are dev-only |

## 1. Architecture & boundary

```
plugins/overseer/
‚îú‚îÄ‚îÄ scripts/            # core ‚Äî pure stdlib, UNTOUCHED except the model add (¬ß2)
‚îú‚îÄ‚îÄ skills/             # core ‚Äî untouched
‚îî‚îÄ‚îÄ dashboard/          # NEW, isolated, optional subtree (its own deps)
    ‚îú‚îÄ‚îÄ backend/        # FastAPI app: serves dist/ + JSON API, shells the CLIs
    ‚îú‚îÄ‚îÄ frontend/       # Vite/React app ‚Üí built to dist/ (committed)
    ‚îú‚îÄ‚îÄ serve.py        # uvicorn launcher (127.0.0.1, opens browser)
    ‚îî‚îÄ‚îÄ pyproject/requirements  # fastapi, uvicorn (+ test deps)
```

**Load-bearing rule:** the backend is a *client of the CLIs*. Reads go through a
new `overseer board --json`; writes through the existing overseer verbs and
`vigil config set`. The backend never imports overseer internals for mutation and
never touches `.workflow/`, so the CLI remains the single writer even though the
board runs in-process.

## 2. Core model additions (pure-stdlib, in `scripts/`)

- `Card.order: int = 0` ‚Äî sort within a column (lower = higher).
- `Card.priority: str | None` ‚Äî `P0`‚Äì`P3`, optional, validated.
- Serialized in frontmatter; `set-field` gains `--order` / `--priority`.
- New **`cli.py board --json`** ‚Äî emits the full board state in one document: every
  card (live + archived) with id/title/status/stage/complexity/budget/parent/
  depends_on/order/priority/sprint, plus computed epic rollups and readiness, plus
  the sprint list. This is the backend's read API. Reuses `relations.py`.

## 3. Backend (`dashboard/backend/`, FastAPI)

Serves the built frontend and a JSON API; every endpoint is a thin shell over a
CLI, returning the refreshed board on mutation:

- `GET /api/board` ‚Üí `overseer board --json` + `vigil context` (`ctx %`) +
  `vigil config get context.threshold`.
- `POST /api/card/{id}/move` (status/stage), `/order`, `/priority`, `/parent`,
  `/depends` (on/off), `/park`, `/unpark` ‚Üí the matching `overseer` verb.
- `POST /api/config/threshold` ‚Üí `vigil config set context.threshold <n>`.

Local-only bind, single user, no auth. Errors from a CLI call surface as a 4xx
with the CLI's stderr; the board never partially-writes (the CLI is atomic per
card).

## 4. Frontend (`dashboard/frontend/`, React + Vite, committed `dist/`)

A kanban board:
- **Columns** by status/stage; **epics** as expandable groups (parent ‚Üí children
  with the rollup line); **dependency** badges (`waiting on WF-003` / `ready`);
  **budgets** with the 2√ó tripwire flag; **parked** and **done** lanes.
- **Interactions:** drag to reorder / move column; click to **expand** a card
  (goal / plan / progress / reviews / verification, read-only prose); inline
  **link** editing (set parent, add/remove deps); **park**/**unpark**;
  **priority** selector; a **threshold** control writing to vigil, with the live
  `ctx %` shown.
- Built to `dist/` and committed ‚Äî running needs only the Python deps.

## 5. Launch, deps, build

- `dashboard/serve.py` starts uvicorn on `127.0.0.1:<port>` (default e.g. 8770)
  and opens the browser. A `commands/dashboard.md` documents it.
- Runtime deps (dashboard-only): `fastapi`, `uvicorn`. Build deps (dev-only):
  node/Vite/React. Documented: `cd dashboard/frontend && npm install && npm run
  build` then commit `dist/`.

## 6. Testing

- Core `order`/`priority` + `board --json`: pytest units (like Spec 1).
- Backend: pytest + FastAPI `TestClient` over a tmp state root ‚Äî `GET /board`
  shape; each mutation invokes the right CLI verb and refreshes; threshold shells
  vigil. (Card writes exercised against a real tmp `.workflow/` via the CLI.)
- Frontend: a light smoke check only; the backend tests + a documented manual
  end-to-end (launch ‚Üí drag ‚Üí expand ‚Üí link ‚Üí set threshold) carry confidence.

## 7. Non-goals

- Auth / multi-user / remote hosting (local only).
- Real-time push (refresh on mutation is enough).
- Replacing `ledger.md` (the board is an *additional* view).
- Touching vigil beyond the threshold config.

## 8. Card breakdown (managed in the overseer ledger)

An **epic** with a linear dependency chain of child cards:

1. **Core model** ‚Äî `order`/`priority` on Card + `set-field --order/--priority` (¬ß2, first half).
2. **`board --json`** ‚Äî the export command (¬ß2, second half). Depends on (1).
3. **Backend** ‚Äî FastAPI app + endpoints + tests (¬ß3). Depends on (2).
4. **Frontend** ‚Äî React/Vite board UI + built `dist/` (¬ß4). Depends on (3).
5. **Launch + docs** ‚Äî `serve.py`, `commands/dashboard.md`, deps/build/README (¬ß5). Depends on (3) and (4).
